# 2026-4-11 exe 打包技术文档

## 1. 目标

新增 `build_exe.py`，用于将当前项目打包为单个 Windows `exe` 文件。生成的程序会内置 Python 运行时和项目依赖，目标是在未安装 Python 的 Windows 环境中也可以直接运行。

默认输出文件为：

```text
dist/eDASread.exe
```

## 2. build_exe.py 的主要功能

`build_exe.py` 主要完成以下工作：

1. 检查打包入口  
   入口脚本固定为项目根目录下的 `run.py`。

2. 检查 PyInstaller 是否已安装  
   如果当前环境缺少 `PyInstaller`，脚本会直接报错并提示安装命令。

3. 清理旧打包产物  
   默认删除旧的 `dist/`、`build/` 和 `{name}.spec`，避免历史缓存干扰。

4. 自动收集运行资源  
   当前打包会一并携带：
   - `resources/`
   - `libs/`
   - `last_params.json`（存在时）

5. 配置源码搜索路径  
   通过 `--paths src` 让 PyInstaller 能正确分析 `src/` 下的模块依赖。

6. 补充隐藏导入  
   显式加入 GUI、存储子包和关键依赖的 `hidden-import`，降低打包后缺模块风险。

7. 生成单文件 exe  
   使用 `--onefile` 模式，最终产物为单个可执行文件。

8. 校验构建结果  
   PyInstaller 完成后，脚本会检查 `dist/eDASread.exe` 是否实际生成，并输出路径和体积。

## 3. 实现方式

### 3.1 打包工具

本方案使用 `PyInstaller`，原因如下：

- 对 Python 桌面 GUI 程序支持成熟
- 支持 `PyQt5`
- 支持 `--onefile`
- 适合当前项目这种“Python + DLL + 资源文件”的结构

### 3.2 打包入口

打包入口为：

```text
run.py
```

`run.py` 会先把 `src/` 加入 `sys.path`，再执行：

```python
from main import main
```

因此围绕 `run.py` 打包即可覆盖整个 GUI 程序。

### 3.3 资源与 DLL 收集

项目运行时依赖以下非 Python 文件：

- `resources/logo.png`
- `libs/wfbg7825_api.dll`
- `last_params.json`

脚本通过 `--add-data` 将这些内容一并打包。

其中：

- `resources/` 用于 GUI Logo
- `libs/` 用于硬件 DLL 加载
- `last_params.json` 用于恢复上次本地参数

### 3.4 模块搜索路径

由于源码位于 `src/` 目录下，而不是根目录平铺结构，因此脚本额外加入：

```text
--paths src
```

这样 PyInstaller 能找到：

- `main.py`
- `main_window.py`
- `wfbg7825_api.py`
- `storage/` 子包

### 3.5 冻结环境兼容处理

仅仅把文件打进 `exe` 还不够，运行时代码也必须能在 PyInstaller 冻结环境下正确找路径。本次同步补充了两处兼容：

1. `src/main_window.py`  
   新增资源根目录解析，冻结运行时优先从 `sys._MEIPASS/resources/logo.png` 读取 Logo。

2. `src/wfbg7825_api.py`  
   在 DLL 搜索逻辑中加入冻结环境路径，优先查找：
   - `sys._MEIPASS/libs/wfbg7825_api.dll`
   - `exe` 同级目录下的 `libs/wfbg7825_api.dll`

没有这一步时，即使 PyInstaller 已把资源和 DLL 带进包里，程序仍可能因为路径不匹配而启动失败。

### 3.6 单文件模式

脚本使用：

```text
--onefile
```

其含义是把 Python 解释器、第三方依赖、项目源码和资源文件统一封装进一个 `exe`，因此目标机不需要单独安装 Python。

### 3.7 窗口模式与控制台模式

默认构建为 GUI 模式：

```text
--windowed
```

这样双击启动时不会额外弹出控制台窗口。

如果需要调试，可使用：

```powershell
python build_exe.py --console
```

此时会保留控制台输出，便于定位启动阶段异常。

## 4. build_exe.py 支持的参数

### 4.1 默认打包

```powershell
python build_exe.py
```

行为：

- 清理旧产物
- 生成单文件 GUI 程序
- 输出到 `dist/eDASread.exe`

### 4.2 控制台调试版

```powershell
python build_exe.py --console
```

行为：

- 生成带控制台窗口的 `exe`
- 适合排查启动异常

### 4.3 仅清理

```powershell
python build_exe.py --clean-only
```

行为：

- 删除旧的 `dist/`、`build/`、`spec`
- 不执行打包

### 4.4 跳过清理

```powershell
python build_exe.py --skip-clean
```

行为：

- 保留当前已有构建目录
- 直接执行本次打包

### 4.5 自定义输出名

```powershell
python build_exe.py --name MyApp
```

行为：

- 输出文件名改为 `dist/MyApp.exe`

## 5. 使用方式

### 5.1 环境准备

建议先安装打包工具：

```powershell
pip install pyinstaller
```

项目运行依赖也应已安装，例如：

```powershell
pip install PyQt5 pyqtgraph numpy psutil
```

### 5.2 执行打包

在项目根目录运行：

```powershell
python build_exe.py
```

打包完成后，在以下位置查看结果：

```text
dist/eDASread.exe
```

### 5.3 分发说明

生成的 `exe` 不要求目标机安装 Python，但仍有以下前提：

- 目标系统为 Windows
- 如果连接真实采集卡，目标机仍需具备对应硬件驱动环境
- 如果只使用仿真模式，不要求安装 Python，也不依赖真实板卡

## 6. 关键实现点

### 6.1 清理逻辑

脚本会清理：

- `dist/`
- `build/`
- `{name}.spec`

这样可以减少旧缓存导致的构建污染。

### 6.2 资源检查逻辑

脚本内部维护固定资源列表，打包前逐项检查存在性。  
其中 `resources/` 和 `libs/` 被视为必需资源；若缺失则直接报错终止。

### 6.3 结果校验逻辑

PyInstaller 执行完成后，脚本会继续检查目标 `exe` 是否真实存在，而不是只依赖命令返回码。

## 7. 当前方案边界

1. 该脚本面向 Windows 打包  
   当前项目依赖 Windows 下的 DLL 和硬件接口，因此本方案主要服务于 Windows 环境。

2. `exe` 不能替代硬件驱动安装  
   它内置了 Python 运行时，但不能替代采集卡驱动、系统权限和设备运行条件。

3. 项目目录结构变化时需要同步更新脚本  
   如果后续调整 `resources/`、`libs/`、`run.py` 等路径，需要同步修改 `build_exe.py`。

## 8. 推荐发布流程

建议按以下流程发布：

1. 在开发机确认 `python run.py` 可以正常启动
2. 执行 `python build_exe.py`
3. 检查 `dist/eDASread.exe` 是否生成
4. 在未安装 Python 的 Windows 测试机验证启动
5. 再进行正式交付

## 9. 结论

`build_exe.py` 已将当前项目的 exe 打包流程脚本化，核心目标是：

- 统一打包入口
- 自动收集 DLL 和资源文件
- 生成单文件 exe
- 支持无 Python 环境运行

该实现可以作为当前项目的标准发布脚本使用。
