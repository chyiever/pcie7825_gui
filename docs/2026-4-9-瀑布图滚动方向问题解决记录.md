# 2026-4-9 瀑布图滚动方向问题解决记录

## 1. 问题概述

本次问题出现在 `Tab2` 的 `Time-Space` 瀑布图优化之后。

现象：

- 优化前：图像按预期从右向左滚动，横轴表示时间，纵轴表示空间/FBG 索引。
- 优化后：图像视觉上变成了从上向下滚动，时间轴和空间轴的显示方向被打乱。

该问题不是采集链路异常，也不是运行时崩溃，而是显示层数据轴解释发生了回归。

从日志看，功能链路本身是正常工作的，`Time-Space` 绘图已成功启用，没有报错：

- `logs/log.txt:813` 记录了 `Time-space plot enabled`

因此问题应定位在 `time_space_plot.py` 的显示矩阵组织方式、`ImageItem` 轴顺序设置，以及 `setImage(...)` 入参之间的匹配关系。

## 2. 相关数据流方向

当前 `Time-Space` 绘图主链路如下：

1. 主窗口把相位数据送入 `TimeSpacePlotWidget.update_data(...)`。
2. 一维数据按采集布局重组为二维矩阵：
   - 代码位置：`src/time_space_plot.py:725`
   - 结果形状：`(fbg_num, frame_num)`
   - 含义：行是空间/FBG，列是时间
3. `_build_display_block(...)` 对二维矩阵做空间范围裁剪、时间/空间降采样：
   - 代码位置：`src/time_space_plot.py:619`
   - 输出仍保持 `(space, time)`
4. `_append_display_block(...)` 把新块写入固定尺寸滚动缓冲区：
   - 代码位置：`src/time_space_plot.py:656`
   - 缓冲区形状固定为 `(space, total_time)`
   - 滚动策略是“左移旧列，右侧追加新时间块”
5. `_update_display(...)` 把缓冲区送给 `ImageItem` 显示：
   - 代码位置：`src/time_space_plot.py:759`

也就是说，优化后的显示缓冲区设计本身是正确的：

- 行方向：空间
- 列方向：时间
- 时间滚动应当表现为横向推进

## 3. 根因分析

根因是同一次优化里，显示层同时做了两件事：

1. 把 `ImageItem` 显式改成了 `row-major`
   - 代码位置：`src/time_space_plot.py:298`
2. 但 `_update_display()` 里仍保留了旧方案中的转置 `.T`

这两件事单独看都可能合理，但组合在一起就出错了。

### 3.1 旧方案为什么需要转置

PyQtGraph 的 `ImageItem` 默认是 `col-major` 语义。历史实现里如果原始数据是 `(space, time)`，为了让横轴显示时间，通常需要传入转置后的 `(time, space)`，或者继续用默认解释方式。

所以旧方案里的 `.T`，本质是在适配旧的轴解释规则。

### 3.2 优化后为什么 `.T` 反而变成错误

优化后代码显式使用了：

```python
pg.ImageItem(axisOrder="row-major")
```

在 `row-major` 下，`ImageItem` 会把输入数组解释为：

- `shape = (height, width)`
- 即第一维是 Y 方向
- 第二维是 X 方向

因此，对于我们的瀑布图：

- 正确输入应当是 `(space, time)`
- 即 `height = space`，`width = time`

这时如果继续执行 `.T`，就会把数据重新变成 `(time, space)`，导致：

- 时间维跑到图像高度方向
- 空间维跑到图像宽度方向
- 虽然坐标轴文本仍写着 `X=Time`、`Y=FBG`
- 但图像本体已经按错误的轴方向被送进去了

最终表现就是：

- 视觉滚动方向从横向滚动变成了纵向堆叠/向下滚动

### 3.3 本次问题的本质

本次问题不是“缓冲区左移逻辑有问题”，而是“显示解释层做了二次转置”。

固定尺寸滚动缓冲区仍然是在按列追加时间数据，这部分逻辑没有错。

真正出错的是最后一步显示：

- 缓冲区内部方向：正确
- `ImageItem` 的 `axisOrder`：已切换为正确的新语义
- `setImage(...)`：却沿用了旧语义下的转置写法

这是一次典型的“局部优化改了接口约定，但旧兼容补丁未同步移除”的回归。

## 4. 解决方案

修复方式很直接：

1. 保留 `ImageItem(axisOrder="row-major")`
2. 去掉 `_update_display()` 中多余的 `.T`
3. 明确注释：显示层统一以 `(space, time)` 传入 `ImageItem`

修复后的关键代码如下：

```python
self.image_item = pg.ImageItem(axisOrder="row-major")
...
self.image_item.setImage(display_data, autoLevels=False)
```

对应代码位置：

- `src/time_space_plot.py:298`
- `src/time_space_plot.py:759`

## 5. 修复后数据方向说明

修复后，数据方向重新统一如下：

- 原始二维相位矩阵：`(fbg_num, frame_num)` = `(space, time)`
- 单次显示块：`(space, time)`
- 固定显示缓冲区：`(space, total_time)`
- `ImageItem(row-major)` 输入：`(space, time)`
- 坐标映射：
  - X 轴：时间
  - Y 轴：空间/FBG 索引

因此滚动行为恢复为：

- 新时间数据追加在右侧
- 窗口满后整体左移
- 用户看到的是从右向左滚动

## 6. 验证结果

### 6.1 日志侧验证

日志中未出现 `Time-Space` 相关异常，说明这不是异常中断，而是纯显示方向错误：

- `logs/log.txt:813`：`Time-space plot enabled`

### 6.2 代码级最小验证

本次修复后，使用最小化离屏验证检查了 `ImageItem` 接收到的图像形状：

- `display_buffer_shape = (3, 10)`
- `image_shape = (3, 2)`
- `bounding = 2 x 3`

这说明：

- 高度方向是空间维 `3`
- 宽度方向是时间维 `2`

即当前显示输入已经恢复为 `(space, time)`，与 `row-major` 的约定一致。

若保留错误的 `.T`，同一组数据会变成 `(2, 3)`，时间维就会被错误送到纵向。

## 7. 影响范围

本次修复只影响 `Tab2` 瀑布图最终显示方向，不影响：

- 采集线程
- 相位数据读取
- `frame_num / fbg_num` 重组逻辑
- 固定尺寸滚动缓冲区左移策略
- 坐标轴文本与时间宽度计算

因此这是一次低风险、定点修复。

## 8. 结论

本次回归的根因是：

- 优化后已切换到 `ImageItem(axisOrder="row-major")`
- 但显示代码仍错误保留了旧方案中的 `.T`

修复后：

- 数据流统一为 `(space, time)`
- `ImageItem` 与显示缓冲区的数据语义重新一致
- 瀑布图恢复为“横轴时间、纵轴空间、从右向左滚动”的正确行为
